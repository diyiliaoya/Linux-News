重构的系统调用入口处理预计将进入 Linux 7.3

发布于 2026 年 7 月 13 日 上午 6:17 EDT。

源于对一项提议的 Linux 内核补丁的审视——该补丁旨在修改 Linux 内核的系统调用编号处理——资深 Linux 内核开发者 Thomas Gleixner 深入研究了内核的系统调用入口处理代码，并进行了大量清理和改进。这一系统调用入口处理的重构现在预计将进入 Linux 7.3 内核周期。

在审查 Linux 系统调用入口处理代码时，他发现 Linux 内核支持的某些 CPU 架构并未以相同方式处理内核栈随机化。一些架构（如 PowerPC）的栈随机化处理存在“错误”且“次优”。

Gleixner 最终调整了所有架构，使其使用通用系统调用入口代码，遵循相同方案，并尽早进行正确的栈随机化。此外，他还修复了代码中的一些“x86 怪异之处”。最终，他的系统调用密集型工作负载基准测试和微基准测试显示，这一重构带来了小幅性能提升。因此，这次系统调用入口处理的重构似乎取得了不错的成果。

Gleixner 详细阐述道：

“那次讨论让我更深入地审视相关代码，像往常一样，发现了许多其他问题。

1) 栈随机化

add_random_kstack_offset() 只能在 enter_from_user_mode() 建立适当状态后调用，因为它会调用可插桩代码。

PowerPC 在这方面处理错误，而其他架构要么在 enter_from_user_mode() 之后调用，要么在 syscall_enter_from_user_mode() 之后调用。

后者是次优的，因为随机化发生在所有用户模式入口工作之后。此外，add_random_kstack_offset() 使用了 get/put_cpu_var()，这使得它可用于可抢占代码，但在中断禁用区域调用时，这会造成无谓的开销。

2) 如上述线程所讨论，仅仅更改 syscall_enter_from_user_mode[_work]() 的函数签名，使其接受一个指向系统调用的指针参数，并在成功时返回 0，这并不直观。此外，这还破坏了跟踪器在将系统调用编号设置为 -1 时的隐式假设。

3) x86 入口代码存在一些历史积累的怪异之处

以下系列通过以下方式解决这些问题：

1) 提供新的 [syscall_]enter_from_user_mode() 变体，这些变体包含栈随机化，并利用新的 add_random_kstack_offset_irqsoff() 变体，避免了 get/put_cpu_var() 开销，并转换了所有使用点

2) 采纳 Jinjie 的 seccomp 补丁，来自：

https://lore.kernel.org/lkml/

并处理反馈（重命名 seccomp 函数）

3) 使 ptrace 和跟踪器相关函数返回布尔值以指示系统调用权限

4) 处理 x86 怪异之处

5) 将整个代码树转换为新方案

这样，所有使用通用系统调用入口代码的架构都遵循相同方案，在正确且尽可能早的位置应用栈随机化，并根据 syscall_enter_from_user_mode[_work]() 的布尔返回值跳过系统调用处理。

应该没有功能变化，至少没有预期中的变化。

在 x86_64 上，系统调用入口代码的最终文本大小比更改前略小。

测试系统调用密集型工作负载和微基准测试显示，总体重构带来了小幅性能提升，但最后一个补丁（旨在使逻辑更易理解）在任一方向上都没有可测量的影响。”

随着这些补丁自昨天起逐步进入 tip/tip.git 的 core/entry Git 分支，预计这些材料将在即将到来的 Linux 7.3 合并窗口期间提交。
