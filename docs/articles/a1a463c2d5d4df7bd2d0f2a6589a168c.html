Reserved THP 特性提议：为 Linux 融合 HugeTLB 与 THP 的优势

Linux 内核开发者、字节跳动工程师 Qi Zheng 发送了一份请求评论（RFC）补丁系列，介绍了一项名为 Reserved THP 的新特性，旨在结合 HugeTLB 和 THP 内核功能的最佳部分。

字节跳动一直在研究 Reserved THP，将其作为在 Linux 内核中统一 HugeTLB 和透明大页（THP）支持以实现内核大页的潜在垫脚石或目标。HugeTLB 具有诸如预留和保证在预留池内分配等优点，但不支持交换。与此同时，THP 不支持预留且分配不保证，但与核心内存管理代码集成更紧密，并能处理交换。

Qi Zheng 在提出 Reserved THP 的想法时解释了自己的经验：

“在我们的内部场景中，由于热升级需求，用户进程需要预留两倍的 HugeTLB 内存。例如，如果进程需要 16GB 的 HugeTLB，则在热升级期间需要额外 16GB 来满足内存分配。升级后，旧进程退出并释放 16GB 的 HugeTLB。因此，在大多数情况下，额外的 16GB HugeTLB 被浪费了。

一个直接的想法是使用 HugeTLB CMA 特性，总共预留 32GB 的 hugetlb_cma。在正常操作期间，消耗 16GB，剩余的 16GB 可由其他进程使用。在热升级期间，我们可以尝试迁移其他进程使用的内存，以分配所需的额外 16GB HugeTLB。这可能有效，但仍需要预留 32GB 内存。

我们还发现，在热升级期间，旧进程的 HugeTLB 中约有 10GB 实际上是冷内存，理论上可以回收。在极端情况下，我们可以只预留 22GB 内存，并在热升级期间回收剩余的 10GB。但不幸的是，HugeTLB 目前不支持交换，而且支持它似乎相当困难。

因此，我们想知道是否可以引入‘reserved THP’，即可以预留的 THP。它可以通过 madvise() 等方法消耗，而正常内存分配无法消耗它。这可以实现类似于 HugeTLB 的效果。而且由于它是 THP，它可以相对容易地支持交换特性，从而完美解决上述问题。”

在过去几年中，内核社区曾讨论过统一 THP 和 HugeTLB，但至今两者仍各自独立。
