Epic Games 构建自有 Git 替代方案以处理大型文件

Epic Games 发布了名为 Lore 的开源版本控制系统，专为处理大型复杂数据设计。该系统使用 Rust 编写，采用 MIT 许可证。Lore 解决了现有工具（如 Git 和 Perforce）在处理大型二进制文件、离线工作和开放规范方面的不足。它通过唯一指纹去重、分块处理大文件、离线提交等功能，为游戏和电影项目提供高效工作流。目前版本为 0.8.3，尚未稳定，接口和存储格式可能变化。

Epic Games 构建自有 Git 替代方案以处理大型文件

Lore 使用 Rust 编写并以 MIT 许可证发布，是一款用于处理大型复杂数据的版本控制系统。

Epic Games 宣布推出 Lore，这是一款公司内部构建并免费发布的开源版本控制系统。

你看，游戏和电影项目的工作流程需要将源代码与大型二进制文件（如构建输入、大数据文件和其他生成内容）混合在一起。问题在于，大多数现有工具并未设计用于很好地处理这种组合。

Git 通过一个名为 Git LFS 的附加组件处理大型二进制文件，而不是将其作为系统的内置部分。

Perforce 能更好地管理二进制文件，但它在日常任务中需要与服务器保持实时连接，并且是一个封闭的专有系统，其他公司无法在其之上构建工具。

Epic Games 表示，现有系统中没有一个能够同时结合二进制文件处理、离线工作和完全开放的规范，这就是他们自行构建的原因。

托管开源项目的 GitHub 替代方案

Lore 会运行一个服务器作为权威节点，控制谁可以访问项目以及如何解决冲突，但日常操作（如保存更改、记录提交或切换分支）完全在本地机器上完成，无需互联网连接。

每个内容都会被赋予一个唯一指纹并仅存储一次，因此相同的数据不会在文件或分支间重复。

它还拥有一个验证系统，可以检查每个修订版本的结构是否被篡改或损坏。大文件被分割成小块，因此编辑一个数 GB 文件的一部分时，无需重新上传整个文件。

此外，默认情况下，你的机器只保存你实际使用的文件，因为 Lore 仅在请求时才会拉取文件数据。

核心库、服务器和 CLI 均已发布，并提供了 JavaScript、Python、C# 和 Go 的官方 SDK。所有内容都通过同一接口路由，因此 CLI 并不是使用 Lore 的专属特权方式。任何使用同一接口构建的工具都可以执行 CLI 能做的所有事情。

该项目尚未达到稳定版本，最新版本为 0.8.3，Epic Games 警告称，接口和存储格式可能在不同版本间发生变化。

你无需安装 Rust 或设置容器即可试用。一个安装脚本即可完成获取 CLI 和服务器二进制文件、将其添加到 PATH 并在本地机器上启动服务器的整个过程。

以下是在 Linux 上配置该脚本的命令：

curl -fsSL https://raw.githubusercontent.com/EpicGames/lore/main/scripts/install.sh | bash -s -- --demo

此外，如果你有任何疑问或只是想讨论相关内容，可以加入一个 Discord 服务器，那里有开发团队和 Lore 社区的成员。
