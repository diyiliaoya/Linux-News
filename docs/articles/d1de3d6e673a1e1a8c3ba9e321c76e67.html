Fedora 45 考虑默认启用 x86_64 Shadow Stack

2026年7月2日 下午5:17 EDT

一项针对 Fedora Linux 45 的变更提案正在考虑中，旨在默认启用 x86_64 架构的 Shadow Stack，以提升现代 Intel 和 AMD 系统的安全性。

该变更提案建议，在 x86_64 架构上，默认对使用 GCC、LLVM Clang 或 Rustc 编译的应用程序/库启用 Shadow Stack 保护。动态链接器或静态启动例程将为所有二进制文件和共享库依赖均支持 Shadow Stack 的进程激活 Shadow Stack。Shadow Stack 由现代 Intel 和 AMD CPU 通过硬件强制实施，有助于抵御面向返回编程（ROP）类型的攻击。

该变更提案进一步阐述：

“此变更在 Fedora Linux 45 上，默认对支持 Shadow Stack 的 x86_64 机器启用 Shadow Stack 保护。动态链接器或静态启动例程将为所有二进制文件和共享库依赖均支持 Shadow Stack 的进程激活 Shadow Stack，从而在可能的情况下默认保护进程。Shadow Stack 是 Intel CET（控制流强制技术）中引入的两项控制流强制功能之一，另一项是间接分支跟踪（IBT），旨在通过保护返回地址来防御面向返回编程（ROP）和面向跳转编程（JOP）攻击。本次 Fedora 变更仅涵盖启用 Shadow Stack 支持。默认启用间接分支跟踪不在本次范围之内。

此变更在大多数情况下向后兼容：自 2018 年起，-fcf-protection 已成为 Fedora 中 redhat-rpm-config 的默认编译时标志，因此大多数二进制文件已使用适当的标记进行构建。因此，应用此变更后，依赖项带有 Shadow Stack 标记的应用程序将透明地获得保护，而在启动时加载任何不符合要求的对象的应用程序将继续在无 Shadow Stack 保护的情况下运行。唯一的新故障模式是，当启用 Shadow Stack 的进程在运行时尝试通过 dlopen 加载不符合要求的共享对象时，会导致 dlopen 错误，错误信息类似于：error: dlopen: /path/to/library.so: rebuild shared object with SHSTK support enabled。”

使用 Shadow Stack 的性能开销通常非常微小甚至不存在，同时能提供更好的系统安全性。这也为在后续 Fedora Linux 版本中启用间接分支跟踪（IBT）以实现完整的控制流强制技术（CET）保护奠定了基础。
