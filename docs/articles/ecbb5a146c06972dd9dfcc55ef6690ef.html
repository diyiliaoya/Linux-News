AMD 畅谈 SPIR-V IR 与 ROCm 的巨大机遇

2026 年 7 月 20 日 美国东部时间下午 12:18

上周发布的 AMD ROCm 7.14 是基于 TheRock 的首个生产版本，但 AMD ROCm 堆栈的未来已更值得期待……特别是 SPIR-V 与 ROCm 的结合成为现实，这将实现跨图形架构/目标的统一二进制文件、更好的 GPU 硬件可移植性，以及针对统一 IR 体验的其他改进。

SPIR-V 是通常与 Vulkan API 驱动程序关联的中间表示，但也得到其他 Khronos API（如 OpenGL 和 OpenCL）的支持。AMD ROCm 一直在努力支持 ROCm 上的 SPIR-V IR 模型，以减少为每个 GPU 目标构建的需求，并提供更统一的体验。但代价是首次启动时 JIT 延迟更高（当首次为特定 GPU 设备/目标调用内核时），并且需要处理包含编译时架构特定代码的变通方案。

ROCm 上的 SPIR-V 已经酝酿了很长时间，也是其 MLIR 策略的一部分。过去几年中，Phoronix 多次报道这一趋势——ROCm 正朝着支持 SPIR-V 的方向发展……

AMD 今天发布了一篇博客文章，概述了 ROCm 中 SPIR-V 支持的当前水平及未来展望。ROCm 7.2 及更高版本已具备基础支持，功能包括 LLVM/Clang 中的 AMD GCN SPIR-V 目标、生产级 SPIRV-LLVM 翻译器路径，以及每进程 JIT 缓存。但未来仍需将树内 LLVM SPIR-V 后端作为默认降级路径来替代翻译器。包级 JIT 安装仍在推进中，同时还需要改进 SPIR-V AMDGCN JIT 路径的调试体验。每架构设备库的 SPIR-V 构建问题也需要解决。

前景令人兴奋：一个 AMD ROCm 二进制文件即可“即插即用”于众多 GPU，实现平坦的构建时间和二进制大小、“免费”的向前兼容性、快速性能以及广泛的库覆盖。ROCm 上的 SPIR-V 堆栈已在 PyTorch 中成功验证。

更多关于 ROCm 上 SPIR-V 堆栈的详细信息，请参阅今天发布的文章。希望我们能在本周晚些时候于加利福尼亚举行的 AMD 推进 AI 活动上听到更多关于其 SPIR-V 工作的消息。
