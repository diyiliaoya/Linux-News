Cryptsetup 2.8.7 发布以应对 Linux 仓促弃用 AF_ALG 的后果

2026年7月21日 下午6:00 EDT

用于配置和管理加密存储设备的 Linux cryptsetup 已推出新版本，开发者正努力应对近期 Linux AF_ALG 接口被弃用的问题。

Cryptsetup 一直是 AF_ALG 的用户之一，该接口允许用户空间应用程序直接访问 Linux 内核内部加密 API。但 AF_ALG 的多个方面因安全问题被迅速移除，AF_ALG 因其“巨大的攻击面”而被弃用。对于 Linux 7.3，AF_ALG 将通过新的 sysctl 可调参数进一步被可选地限制。这给 Cryptsetup 等用户空间软件带来了挑战，因为它们长期依赖这个如今被认为安全负担过大的接口。

随着今天 Cryptsetup 2.8.7 的发布，他们已开始调整代码，确保其能在没有 AF_ALG 的情况下生存，并能良好地回退到用户空间加密库。目前用户空间库方法存在一些限制，如 v2.8.7 所述：

“与 Linux 内核 AF_ALG 加密用户空间接口弃用相关的变更：Linux 内核维护者决定弃用 AF_ALG 接口（cryptsetup 大量使用该接口），并计划出于安全原因将其移除（或严格限制）。Libcryptsetup 使用用户空间（主要通过 AF_ALG）处理 LUKS 密钥槽以及其他格式的磁盘元数据。使用 AF_ALG 确保了用户空间和后续内核设备激活时使用相同的算法集。

虽然 libcryptsetup 有回退到用户空间库的概念，但并非在所有情况下都使用。

在此版本中，libcryptsetup 可以使用用户空间加密库、AF_ALG（如果存在），并在某些情况下（如 LUKS 密钥槽）回退到临时 dm-crypt 映射。最后一个选项需要 root 权限。这确保了即使 AF_ALG 被禁用或受限，大多数操作仍能继续工作。

不幸的是，移除 AF_ALG 可能会导致严重的兼容性问题，如果所需算法（或加密模式）未在用户空间库中实现。一个典型例子是 Adiantum 密码，它仅在内核中实现。此外，像 Serpent 或 Twofish（在 XTS 模式下）等密码在多个用户空间库中缺失。如果 AF_ALG 接口不可用，cryptsetup 的密码基准测试将不再可用。”

除了应对 AF_ALG 的快速消失，Cryptsetup 2.8.7 还带来了密钥环处理变更、处理可能的 LUKS 卷密钥摘要冲突的变更、libgcrypt 后端中的 Aria 和 Camellia 密码支持，以及其他修复。
